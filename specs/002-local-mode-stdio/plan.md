# Implementation Plan: Local Deployment Mode with stdio MCP Server Support

**Branch**: `002-local-mode-stdio` | **Date**: 2026-05-23 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-local-mode-stdio/spec.md`

## Summary

Add a local deployment mode that lets a developer run the proxy on their laptop with an embedded SQLite database — no Postgres, no cloud services, no Docker required. In local mode the user is their own admin, OAuth2 app credentials are prohibited (PAT/API key only), and stdio-based MCP servers can be registered in the catalog. A `mcp.CommandTransport` from the go-sdk manages child-process lifecycle for stdio servers. A thin interface layer over the existing store code enables plugging in either the existing Postgres backend or the new SQLite backend at startup.

## Technical Context

**Language/Version**: Go 1.26 (existing codebase)

**Primary Dependencies**:
- Existing: `github.com/modelcontextprotocol/go-sdk`, `github.com/jackc/pgx/v5`, `golang.org/x/oauth2`
- New: `modernc.org/sqlite` (pure-Go SQLite, no CGO) for local mode storage

**Storage**:
- Hosted mode: PostgreSQL via pgx (unchanged)
- Local mode: SQLite via `database/sql` + `modernc.org/sqlite`

**Testing**: `go test ./...` (unit); `go test -tags integration ./tests/integration/...` (Postgres only)

**Target Platform**: Linux (Cloud Run / Docker) for hosted; macOS/Linux for local

**Project Type**: Web service / single binary

**Performance Goals**: Local mode — tool call overhead <100ms; stdio server startup <2s per SC-002

**Constraints**: Local mode binary MUST work with zero external services; single `go build` command; no CGO

**Scale/Scope**: Local = single user; hosted = existing multi-tenant scale

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| Principle II: local execution | This feature IS local deployment mode — the spec explicitly requires it. The dual-proxy model (local + hosted) extends the proxy's value to personal use without deprecating the cloud offering. | Keeping cloud-only would make stdio servers impossible and exclude personal/offline use cases. The constitution was written before this use case was defined. |
| SQLite embedded database | Local mode requires zero-dependency storage. SQLite is the universal embedded relational database. | Postgres requires Docker/cloud — contradicts "run with a single binary." BoltDB/BadgerDB require rewriting all SQL. |
| Store interface layer | Needed to plug in SQLite and Postgres under the same handler code. | Without interfaces, every handler would need a mode-branch, which is worse. Three lines per store type is the minimal abstraction. |

**Constitution amendment required**: Principle II must be updated in this PR to recognize local deployment mode as a supported target. See amendment notes in `.specify/memory/constitution.md` (to be updated in tasks).
